# System Audit Report — Moldati Owner

_Adversarial end-to-end audit before production. Method: ~22 specialized reviewer agents (high reasoning effort) across the Flutter app, owner/admin panels, backend, sync layer, SQLite, and online sync; **every finding was independently cross-verified by two skeptic agents** (a correctness lens + a reproducibility lens). A finding is **Confirmed** only when BOTH skeptics agreed it is real; **Potential** when one did; **Needs-validation** when verification was cut short by an API session limit; **Refuted** when both skeptics dismissed it._

## Executive summary

The audit produced **83 distinct findings**: **74 confirmed**, **1 potential**, **4 needs-validation** (verification truncated by a session limit — includes 2 critical), and **4 refuted**.

Confirmed by severity: **Critical 3 · High 35 · Medium 27 · Low 9**.

The findings cluster into a few systemic root causes that should be addressed **before multi-device / multi-accountant production use**:

1. **No conflict resolution — unconditional last-writer-wins with no `updatedAt`/version compare** (both `syncController.push` server-side and `SyncService.pull` client-side). Two devices, or one device after an offline spell, silently destroy each other's edits (lost updates), and `pull()` overwrites local writes that land between its push and pull steps. This is the single largest data-integrity risk.
2. **Receipt numbers are minted client-side as per-branch `MAX+1` with no uniqueness guarantee.** Two devices/accountants offline mint the *same* `receipt_no`; both survive the sync → duplicate, irreconcilable financial records.
3. **Logout-wipe can permanently destroy data** — when offline, when the plan disables sync, or when the pre-wipe push fails (the v8 "logout clears local data" behavior has no successful-push/online guard).
4. **Security gaps that block production**: a hard-coded fallback `JWT_SECRET`, default `admin/admin123` bootstrap creds, subscription `expiresAt` never enforced, `maxDevices` bypassable by omitting the device payload, and `/api/sync/push` trusting whole-row data with **no validation or server-side branch/owner/permission scoping** (any accountant can write any entity/branch into the owner mirror).
5. **Financial inconsistency from the v8 discount** — the backend dashboard subtracts the waived discount from "remaining" but the **app Dashboard and Reports do not**, so the four "remaining/paid" computations disagree.
6. **Scale/longevity**: Dashboard, Reports and the Paid/Unpaid screen load the **entire** subscriber table (`limit:10000`) every refresh; `pull()` re-applies the **whole** mirror (the `since` watermark is computed but never sent); and hot queries hit **un-indexed** columns.

> Verification was interrupted by an API session limit (resets 13:40 Asia/Riyadh), so **4 findings (incl. 2 critical / 1 high) are listed under "Needs validation"** — they were flagged by reviewers but did not receive two independent confirmations. Re-run the verification pass after the reset to promote/clear them.

## Confirmed issues

### Critical
#### Server apply is unconditional last-writer-wins with NO updatedAt/version compare — a stale offline device silently destroys a newer edit (lost update)
- **Area:** `conflict-resolution` &nbsp;|&nbsp; **Severity:** CRITICAL
- **Location:** backend/src/controllers/syncController.js:37-41; lib/core/sync_service.dart:53,71
- **Scenario:** Subscriber S (uuid X), amps=10. Device A and Device B both have S synced.
T0: Device B goes offline. Operator edits S on B: amps=10 -> amps=30 (B's outbox now holds an UPDATE for X). B stays offline for an hour.
T1: On Device A (online) operator edits the SAME subscriber X: name fix + amps stays 30 conceptually... actually A sets amps=25 and phone='555'. A pushes immediately: server mirror row X = {amps:25, phone:'555', updatedAt = A's push time T1}.
T2: Device B regains connectivity. SyncController.maybeAutoSync -> push(). push() reads the CURRENT local row X on B (amps=30, OLD phone=null) and sends {data: full B row, updatedAt: DateTime.now() = T2 (push time, > T1)}.
T3: Server findOneAndUpdate (syncController.js:37) does an UNCONDITIONAL $set of data/deleted/updatedAt. Mirror row X becomes B's stale snapshot: amps=30, phone=null. A's phone='555' and amps=25 are gone from the mirror.
T4: Any new device, or A after a 'delete-local + pull' / branch-switch, pulls the mirror and gets B's stale row. A's edit is permanently lost. The admin panel also shows the stale data.
- **Root cause:** The mirror has no conflict detection. (1) On the device, push() stamps updatedAt = DateTime.now().toUtc() at PUSH time (sync_service.dart:53,71), NOT the row's real modification time — so a row edited an hour ago but pushed now looks 'newer' than a row genuinely edited 30 min ago. (2) The server (syncController.js:37-41) does findOneAndUpdate with a plain $set and no guard (no `updatedAt: {$lt: rec.updatedAt}` filter, no version field), so whichever push arrives LAST wins regardless of which edit actually happened later. There is no per-field merge and no rejection of older writes.
- **Recommended fix:** Make conflict resolution real: (a) store a true per-row mtime on each business row (e.g. an `updated_at` column maintained by the UPDATE triggers) and send THAT as updatedAt instead of the push-time clock; (b) on the server, only overwrite when the incoming updatedAt is newer: add `updatedAt: { $lt: incoming }` (or a monotonically increasing version) to the findOneAndUpdate filter, and upsert-only-if-newer, so an older snapshot can never clobber a newer one. Field-level merge would be even safer for independent-field edits.

#### Offline user-initiated logout wipes ALL local data including unpushed offline edits (permanent loss)
- **Area:** `logout-login` &nbsp;|&nbsp; **Severity:** CRITICAL
- **Location:** lib/controllers/auth_controller.dart:474-482; lib/controllers/sync_controller.dart:119-159; lib/views/screens/home_screen.dart:23; lib/views/screens/settings_screen.dart:492
- **Scenario:** Owner works offline (no internet, e.g. at the generator site) and records the month's payments / adds subscribers — these sit in sync_outbox, never pushed to the mirror. The user taps the logout icon (home AppBar or Settings). logout(wipeLocal:true) runs: it calls SyncController.syncNow(silent:true), which at sync_controller.dart:123-127 detects offline, sets lastSyncError='network' and RETURNS NORMALLY (no throw). Control returns to auth_controller.dart:478 which unconditionally calls sync.deleteLocalData(silent:true), wiping the 7 business tables + the outbox (sync_service.dart:137-146). The unpushed edits existed ONLY locally, so they are gone forever; the server mirror never received them. There is no confirmation dialog and no connectivity check on either logout button.
- **Root cause:** logout-wipe treats the pre-wipe push as 'best-effort' but then wipes unconditionally. syncNow() does not throw when offline (or when push fails) — it swallows the failure into lastSyncError — so the try/catch around it in logout() never prevents the destructive deleteLocalData() from running. The design comment ('push any unsynced changes up first so they aren't lost') is not actually enforced.
- **Recommended fix:** Before wiping, verify the push truly succeeded and the outbox is empty. E.g. in logout(): only call deleteLocalData() if pendingCount()==0 after the push attempt; if offline or push failed (outbox still non-empty), skip the wipe (and ideally block/confirm the offline logout in the UI). Make syncNow signal success/failure (return bool) so the wipe is gated on it. Also add a connectivity check + confirmation dialog on the logout buttons (home_screen.dart:23, settings_screen.dart:492).

#### Logout-wipe on a sync-disabled plan destroys all data with no possible recovery
- **Area:** `logout-login` &nbsp;|&nbsp; **Severity:** CRITICAL
- **Location:** lib/controllers/sync_controller.dart:119-120,166-167,277-290; lib/controllers/auth_controller.dart:474-482,316-322
- **Scenario:** An account on a plan where canSync is false runs in 'offline-only' mode (local SQLite is the only store; the app never pushes/pulls). The user taps logout. logout(wipeLocal:true) -> syncNow(silent:true) hits sync_controller.dart:120 (`if (!_canSync) return;`) and pushes NOTHING. Then deleteLocalData(silent:true) wipes everything. On the next login, the owner post-login pull (auth_controller.dart:317-319 -> SyncController.pull) hits sync_controller.dart:167 (`if (!_canSync) return;`) and is a no-op. Result: the entire business database is permanently destroyed and cannot be restored from the mirror (there is no mirror for this plan).
- **Root cause:** The logout-wipe path assumes 'the next login auto-pulls the data fresh from the backend', but for canSync==false there is no server mirror and both push and pull are no-ops. The wipe is unconditional regardless of whether a recoverable server copy exists.
- **Recommended fix:** Never wipe local data when canSync is false (no server copy to restore from). Gate the wipe in logout() on auth.canSync && a confirmed-successful push, otherwise fall back to a plain logout that keeps local data.

### High
#### App dashboard 'Remaining Fees' does NOT subtract the waived discount, so it disagrees with the backend owner panel and contradicts its own paid/unpaid counts
- **Area:** `discount` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** lib/controllers/dashboard_controller.dart:95
- **Scenario:** Single-branch owner, month with standard price 5000. Three 10A subscribers (due 50000 each, expected total 150000): sub-disc pays 30000 cash + 20000 value-discount (full, fully covered), sub-plain pays 50000, sub-unpaid pays 10000. After collection the app dashboard reloads loadStats(): expected=150000; getCollectedSum returns Σ paid_amount = 30000+50000+10000 = 90000 (discount correctly excluded); totalDue = expected - collected = 150000 - 90000 = 60000. Meanwhile countByPaymentStatus reports paidCount=2 (sub-disc + sub-plain), unpaidCount=1 (sub-unpaid owes 40000). So the UI shows 'Remaining 60000' while only 40000 is actually owed by the single unpaid subscriber — the 20000 waived discount is shown as still-owed AND the subscriber is simultaneously marked fully paid. The exact same data on the owner panel shows Remaining 40000 (backend buildDashboard subtracts discountTotal at accountController.js:169). Two screens of the same product disagree by Σ discount_value for every discounted subscriber.
- **Root cause:** P5 (spec.md:47) requires remaining = expected − collected − Σ discount_value in lockstep with the backend. The backend was updated (accountController.js:128-169 accumulates discountTotal and does `remaining = expected - collected - discountTotal`), and the coverage/paid-unpaid path was updated (getByPaymentStatus folds discount_value into coverage, getDueAmount adds r.discountValue). But the app's aggregate 'Remaining' figure was missed: dashboard_controller.dart:95 still does `totalDue = expected - totalCollected` with no discount term. getCollectedSum is (correctly) Σ paid_amount only, so the waived discount never gets removed from the due side on the app, leaving it double-represented (waived as coverage for paid/unpaid, yet still owed in the remaining total).
- **Recommended fix:** Compute the in-scope Σ discount_value for the month/branch (e.g. add a ReceiptRepository.getDiscountSum(month, {accountantId, branchId}) mirroring getCollectedSum but summing IFNULL(discount_value,0) over status='valid' rows) and change line 95 to `totalDue.value = expected - totalCollected.value - discountSum;` so it matches accountController.js:169 and the paid/unpaid counts. Keep netProfit/collected on Σ paid_amount only (already correct).

#### Reports screen 'Remaining' total also omits the waived discount (same lockstep break as the dashboard)
- **Area:** `discount` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** lib/controllers/reports_controller.dart:119
- **Scenario:** Same data as above viewed on the Reports screen for the month. expectedTotal=150000, collectedTotal=getCollectedSum=90000, remainingTotal = 150000 - 90000 = 60000, while paidCount=2/unpaidCount=1 (only 40000 truly owed) and the owner-panel reports view derives remaining = 40000. The printed/exported report and the backend mirror disagree by the discount amount whenever any subscriber was given a full-payment discount.
- **Root cause:** Identical omission to dashboard: reports_controller.dart:119 does `remainingTotal = expectedTotal - collectedTotal` with no `- Σ discount_value` term. collectedTotal deliberately excludes discount (correct for revenue/netProfit at line 122), but the remaining/due side then never accounts for the waived coverage, contradicting getByPaymentStatus and the backend's `expected - collected - discountTotal`.
- **Recommended fix:** Subtract the same in-scope Σ discount_value (scope=accountantId, branch=branchId, to match collectedTotal's scope) from remainingTotal: `remainingTotal.value = expectedTotal.value - collectedTotal.value - discountSum;`. Reuse the new getDiscountSum helper so dashboard and reports stay in lockstep with each other and with accountController.js:169.

#### Edit-after-delete across devices resurrects deleted rows (no tombstone protection / no LWW guard on push)
- **Area:** `data-corruption-traces` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** backend/src/controllers/syncController.js:37-41; lib/core/sync_service.dart:31-89
- **Scenario:** Device A and Device B both have subscriber S (uuid X). Day 1: A (online) deletes S. The DELETE trigger enqueues op='delete' for X; A.push() sends {entity:'subscribers', localId:X, deleted:true} and the server SyncRecord for X is set deleted=true. Device B was offline and never saw the delete. Day 2: B (still has S locally) edits S's name; the UPDATE trigger enqueues op='upsert'; when B comes online B.push() sends {localId:X, deleted:false, data:{...full row...}}. The server's findOneAndUpdate unconditionally $sets deleted=false + data (no comparison of updatedAt, no 'is this already a tombstone?' check). The row is resurrected on the server. On A's next pull (always a full pull, since=null), A re-inserts S via ConflictAlgorithm.replace. S is back from the dead on every device. The same trace deletes a board on A then collects a receipt on B for a subscriber under that board: the board tombstone is overwritten and the board reappears.
- **Root cause:** The mirror is pure last-writer-wins by PUSH ORDER, not by edit time. push() stamps updatedAt = DateTime.now() at upload time (sync_service.dart:53), not at the moment the change happened, and the server upsert (syncController.js:37-41) never compares the incoming updatedAt against the stored one nor refuses to clear an existing tombstone. A delete is just another writable field, so any later upsert from a device that missed the delete revives the row.
- **Recommended fix:** Make the server upsert last-write-wins by the DEVICE CHANGE TIME and protect tombstones: (a) capture the real change timestamp in sync_outbox (the trigger already writes ts) and send that as updatedAt instead of now(); (b) in syncController.push, only apply a record when incoming.updatedAt > stored.updatedAt (conditional findOneAndUpdate with an updatedAt filter), so a stale upsert can't overwrite a newer delete. Deletes should also be sticky for a grace window. This is a mirror-layer fix, not an outbox/trigger change.

#### Same receipt_no minted on two devices for the same branch (duplicate sequential numbers survive sync)
- **Area:** `data-corruption-traces` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** lib/data/repositories/billing_repositories.dart:132-142; lib/controllers/billing_controller.dart:238-265
- **Scenario:** Branch B currently has max receipt_no = 100. Device A (offline) and Device B (offline) each collect a payment. Both call getNextReceiptNumber(branchId:B) → both read MAX(receipt_no)=100 → both assign receipt_no=101. Each generates a DIFFERENT uuid (Uuid().v4()), so the receipts table PK does not collide and both inserts succeed locally. Both push: server stores two SyncRecords (distinct localId), neither overwrites the other. Every device then pulls both → two valid receipts with receipt_no=101 in the same branch. Reports, printed invoices, and any 'find receipt #101' lookups now return two different receipts; getNextReceiptNumber keeps returning 102 next, so the duplicate is permanent.
- **Root cause:** Per-branch receipt numbering is computed as a local MAX+1 with no central allocator and no uniqueness constraint (uuid is the PK; receipt_no has no unique index). Concurrent offline collection on two devices is a normal multi-device/multi-accountant workflow, so the collision is expected, not exotic.
- **Recommended fix:** receipt_no cannot be both gap-free-sequential and offline-multi-device-safe. Either (a) make the human-facing number a composite that embeds a device/installId discriminator, or (b) treat receipt_no as display-only and key everything on uuid, accepting that numbers may duplicate across devices and surfacing it, or (c) allocate ranges per device. At minimum, dedupe/renumber on pull so two receipts never share (branch_id, receipt_no).

#### Per-accountant permissions are enforced only in the Flutter UI; the sync push endpoint lets any accountant write every entity into the owner's mirror
- **Area:** `auth-roles` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** backend/src/controllers/syncController.js:15-46 (push); lib/core/permissions.dart:1-37; lib/controllers/auth_controller.dart:80-85
- **Scenario:** Owner creates accountant 'cashier' with permissions:[] (meant only to record payments + print receipts — the always-allowed core job, no subscribers/boards/expenses/prices rights). cashier logs in (role 'accountant'), gets a valid JWT. Using any non-stock client (curl, a patched APK, or replaying the app's own /api/sync/push), cashier POSTs records for entity 'subscribers' (delete tombstones), 'boards', 'monthly_prices' (change price_per_amp), or 'expenses'. requireFeature('sync') passes (resolved via the owner's plan). push() does NO check of the accountant's `permissions` — it upserts every record into effectiveOwnerId(owner)'s SyncRecord mirror. When the owner (or another accountant) later pulls on a new/cleared device, those malicious rows are written into the real local SQLite DB. The Perm.subscribers/boards/expenses/prices gating in lib/core/permissions.dart exists ONLY in the app's `can()` UI checks; the server trusts the accountant for all entities.
- **Root cause:** Authorization for the four grantable permissions is implemented purely client-side (auth_controller.can / screen Obx gates). The backend sync mirror authenticates the accountant and scopes by effective owner but never validates the pushed entity against the accountant's permission set, so the security boundary is only cosmetic for a tampered or replayed client.
- **Recommended fix:** Enforce permissions server-side in push(): for an accountant (req.user.role==='accountant'), map each record.entity to its required Perm (subscribers->subscribers, boards/circuits->boards, expenses->expenses, monthly_prices->prices; receipts/branches/accountants handled by their own rules) and reject (403) records the accountant's req.user.permissions does not include. Always-allowed receipts can stay ungated. Treat the accountant permission list as the authoritative gate, not just the UI.

#### Schema-drift on pull aborts the ENTIRE transaction (all-or-nothing), so a newer device's records make an older device pull nothing
- **Area:** `sync-pull` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** lib/core/sync_service.dart:101-128 (insert at 118-122)
- **Scenario:** Device A runs app v8 (SQLite schema v7: receipts has discount_type, discount_value, discount_amps). A collects a receipt and pushes it; the server mirror stores the raw row including those columns. Device B is still on an older build (schema v6, receipts has no discount_* columns). B pulls: pull() loops the server records and calls txn.insert('receipts', data) with data containing discount_type/discount_value/discount_amps. SQLite raises 'table receipts has no column named discount_type'. Because the whole apply runs inside one db.transaction((txn) async {...}), that single bad record throws and the ENTIRE transaction rolls back. Result: B applies ZERO records on this pull (not even the boards/circuits/subscribers that appear earlier in the list and inserted fine) and the controller catch sets _pullFailed -> 'sync_failed'. Every subsequent full pull (since is never used, see other finding) re-fetches the same poisoned record and keeps failing, so B can never receive ANY data while a newer-schema device exists in the account.
- **Root cause:** pull() applies the server snapshot inside a single transaction and inserts the server `data` map verbatim with ConflictAlgorithm.replace, never filtering to columns the local table actually has. Older->newer works because new columns have DEFAULTs; newer->older throws on the unknown column, and the all-or-nothing transaction turns one incompatible row into a total pull failure.
- **Recommended fix:** Apply each record defensively: read the local table's columns once (PRAGMA table_info per entity) and drop keys not present in the schema before txn.insert, OR wrap each record apply in its own try/catch (or savepoint) so one incompatible/corrupt row is skipped instead of failing the whole pull. Filtering unknown columns is the correct behavior for a push-only mirror that can hold rows from a newer schema.

#### Branch-switch wipes local data then pulls; a pull that throws (e.g. schema-drift above) strands the device with an EMPTY database
- **Area:** `sync-pull` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** lib/controllers/sync_controller.dart:245-265
- **Scenario:** Owner taps a different branch. switchBranch() does: _sync.push() (ok), _sync.deleteLocalData() (wipes all 7 business tables locally), then _sync.pull() inside a try with exactly ONE retry. If pull() throws for ANY reason after the wipe -- a transient network error mid-apply, OR the schema-drift insert error which rolls back the whole transaction and re-throws -- both the first call and the single retry throw, the outer catch runs branchCtrl.setBranch(target) and shows 'sync_failed'. Local data is already gone and nothing was restored, so the user is left looking at a completely empty branch (no boards/subscribers/receipts) until some future pull happens to succeed. The server mirror is intact but unreachable to the UI.
- **Root cause:** The destructive deleteLocalData() is committed before the restoring pull is known to succeed, and pull()'s transaction is all-or-nothing, so any single failure leaves the device with no data. A one-shot retry of the same deterministic failure (schema drift) does not help.
- **Recommended fix:** Make the clear+pull atomic-ish: only clear after a successful pull (pull into a staging area or detect success first), or at minimum do not call deleteLocalData() until _sync.pull() has returned successfully. Combined with per-record skip in pull() so a single bad row cannot fail the restore.

#### Legacy NULL branch_id rows pulled onto a fresh device are invisible in the Main-branch view yet still occupy their circuits (IFNULL vs plain branch_id mismatch)
- **Area:** `branch-isolation` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** lib/data/repositories/core_repositories.dart:101-104 (boards getAll), 205-208 (circuits getByBoardId), 373-376 (subscribers getAll branch clause), 398-407 (getByCircuit), 413-425 (getByBoard) vs 278-281 (isCircuitTaken IFNULL), 322-325 (takenCircuitIds IFNULL), 459-461 (getByPaymentStatus outer s.branch_id = ?)
- **Scenario:** Timeline: Owner device A runs an old schema (v2..v4 - sync change-capture exists since v2, but the branch_id column does not). The sync triggers push board/circuit/subscriber/receipt rows to the server mirror with NO branch_id key (the column doesn't exist, so push()'s db.query at sync_service.dart:64 returns rows without it; backend stores `data` verbatim, syncController.js:33-34). The owner later installs the app on a NEW device B (fresh v7 schema) and does pull-latest. SyncService.pull (sync_service.dart:116-123) inserts those rows verbatim, so on device B they land with branch_id = NULL. The v5 backfill UPDATE (db_helper.dart:209) only runs during _onUpgrade - it NEVER touches rows that arrived via pull on a clean install. On device B the active branch is Main (scopeBranchId = 'main'). loadBoards/loadSubscribers/loadCircuits query `branch_id = 'main'`, which excludes NULL, so the entire legacy customer base is INVISIBLE on device B. Meanwhile isCircuitTaken (278-281) and takenCircuitIds (322-325) use IFNULL(branch_id,'main'), so those same hidden subscribers still report their circuits as taken: the owner sees an empty circuit but gets ValidationException('circuit_in_use') when adding a subscriber to it. Same split for nameExists (121-124, 296-302): a 'duplicate_name' rejection against an invisible subscriber.
- **Root cause:** Read/visibility queries scope with plain `branch_id = ?` while validation/due queries scope with `IFNULL(branch_id, kMainBranchId) = ?`. The two interpretations of a NULL branch_id diverge, and NULL rows are reachable on any clean-install device via pull (the v5 backfill is upgrade-only and is never applied to pulled rows).
- **Recommended fix:** Make the NULL-branch interpretation uniform: either (a) have list/count reads also use IFNULL(branch_id,'<kMainBranchId>') = ? (matching isCircuitTaken/nameExists/getByPaymentStatus), or (b) run a one-time idempotent backfill UPDATE branch_id='main' WHERE branch_id IS NULL after every pull (not just during _onUpgrade). Option (a) is the minimal consistent fix and aligns all branch reads.

#### switchBranch can strand the device with an empty local DB (data wiped, both pull attempts fail, no auto-recovery)
- **Area:** `branch-isolation` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** lib/controllers/sync_controller.dart:236-271
- **Scenario:** Owner taps to switch branches while online. switchBranch sets isPulling=true, shows the overlay, push() succeeds, then deleteLocalData() WIPES all 7 business tables (sync_service.dart:137-146). It then calls pull(); if it throws, the inner catch (252-254) retries pull() once. If the retry ALSO throws (connectivity dropped right after the clear, server 502, or a mid-pull socket reset - all realistic on mobile), the exception propagates to the OUTER catch (261), which only does branchCtrl.setBranch(target) plus a 'sync_failed' snackbar. The local DB is now EMPTY and the chosen branch is active. Auto-sync only PUSHES, never auto-pulls (maybeAutoSync at 78-91 calls syncNow, not pull), so nothing restores the data automatically. The owner is left with an empty dashboard/lists until they manually find and tap the pull-latest button; with no network for a while the business appears to have lost ALL its data. The server mirror is intact so it is recoverable, but for an offline-first app this is effectively an on-device outage with no automatic recovery.
- **Root cause:** deleteLocalData() runs BEFORE a successful pull is guaranteed; the failure path has no rollback and no automatic re-pull, and the 30s heartbeat never pulls. The single retry does not cover a sustained outage that begins after the destructive clear.
- **Recommended fix:** Pull into a staging area first and only clear+swap once the pull succeeds; OR set a persistent 'pending_branch_pull' flag when pull fails after the wipe and have maybeAutoSync()/connectivity-regain trigger a pull until it succeeds, instead of relying on a manual tap.

#### push() trusts whole-row data with zero validation: a client can write arbitrary entities/fields, cross-branch and cross-accountant rows into the owner mirror
- **Area:** `backend-sync-ctrl` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** backend/src/controllers/syncController.js:26-43
- **Scenario:** Owner O has accountant ACC confined to branchId='B-2' (User.branchId, see backend/src/models/User.js:58 and the accountant money/branch scoping in backend/src/controllers/accountController.js:69,120). ACC signs in (auth_controller.dart:501 _onAccountantLoggedIn -> SyncController.pull). effectiveOwnerId(req.user) resolves to O (backend/src/utils/effectiveOwner.js:14). ACC's device (or a tampered/old client) POSTs /api/sync/push with records=[{entity:'receipts', localId:'r1', deleted:false, updatedAt:..., data:{uuid:'r1', branch_id:'B-1', accountant_id:'<some-other-accountant-id>', paid_amount:99999, ...}}]. The handler stores rec.data verbatim (line 34, data = rec.data) into O's mirror under (user=O, entity='receipts', localId='r1'). Nothing checks: (a) entity is one of the known 8 tables, (b) data.branch_id equals ACC's assigned branchId, (c) data.accountant_id equals ACC's own id, (d) localId === data[pk]. The owner's admin views (adminController.js:236-238 filters on data.branch_id) and the accountant performance dashboard (accountController.js:120 inScope keys on data.accountant_id) now show a receipt in branch B-1 attributed to a different accountant with a fabricated amount. Branch isolation and accountant attribution are enforced only by trusting fields the unprivileged client supplies.
- **Root cause:** The mirror is push-only and the handler was written to 'store the raw SQLite row as-is' (comment line 33). It validates only that entity and localId are strings (line 27). It never (1) whitelists entity against the known synced tables, (2) for accountants stamps/validates data.branch_id against req.user.branchId, or (3) stamps/validates data.accountant_id against the authenticated accountant's id. effectiveOwnerId collapses an accountant into the owner with no per-accountant/per-branch constraint, so an accountant has the same unrestricted write surface as the owner over the whole account's data.

#### Edit-vs-delete tombstone race: deleting a row on one device resurrects or destroys a concurrent edit because tombstones win unconditionally and have no causality check
- **Area:** `conflict-resolution` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** backend/src/controllers/syncController.js:31,37-41; lib/core/sync_service.dart:62-73,112-114
- **Scenario:** DELETE-beats-EDIT (data destroyed): Subscriber X exists on Device A and Device B.
T0: B offline. On A, operator deletes X (board reorg). A pushes -> mirror row X: {deleted:true}.
T1: On B (still offline) operator edits X: amps -> 50, marks status active. B's outbox holds an UPDATE for X.
T2: B comes online, push(): the LOCAL row X still exists on B, so push sends {deleted:false, data: B's full row, updatedAt = T2}. Server unconditionally $sets deleted:false + B's data (syncController.js:37). X is RESURRECTED in the mirror with B's edit. On A's next pull X reappears. Whether that is correct is undecidable, but the system silently picks 'last push wins' with no rule.
EDIT-beats-DELETE (edit lost): swap the order. T0: A edits X (amps 50) and pushes -> mirror {deleted:false, amps:50}. T1: B (had X) deletes X offline; B's outbox holds a DELETE. T2: B pushes: push() sees op=='delete', sends {deleted:true} (sync_service.dart:62-73). Server $sets deleted:true unconditionally. A's amps=50 edit is gone. Next time A does delete-local+pull, pull() applies the tombstone (sync_service.dart:112-114) and deletes X locally on A too. The edit is destroyed on every device.
- **Root cause:** Tombstones are treated as plain last-writer-wins like any other field (syncController.js sets `deleted` via the same unconditional $set). There is no causality/version tracking, so the engine cannot tell whether an edit happened before or after a delete; it just applies whichever push lands last. On the device side pull() turns any tombstone into an unconditional local delete (sync_service.dart:112-114) with no check for un-pushed local edits to that same row.
- **Recommended fix:** Same root fix as the lost-update finding: attach a real monotonic version/mtime per row and only apply a delete or an edit if it is causally newer than the stored state. At minimum, on the device, pull() must not delete a local row that has a pending outbox UPDATE entry (locally-pending edit beats an incoming tombstone until resolved), and the server must reject a delete/edit older than the stored updatedAt.

- **Location:** backend/src/config/env.js:27 (used by backend/src/utils/token.js:9,14; backend/src/middleware/auth.js:22)
- **Scenario:** The production deployment ships with the Flutter app pointed at https://generator.tikritstore.shop by default (CLAUDE.md / lib/core/api_config.dart). If the production host is started without JWT_SECRET set in its environment (e.g. .env missing/incomplete — note .env is gitignored and not present in the repo, so this is the from-scratch state), env.JWT_SECRET resolves to the literal 'change-me-in-production-please', which is committed in env.js and .env.example. An attacker who reads the public GitHub source signs their own JWT: jwt.sign({sub:'<any user _id>', role:'admin'}, 'change-me-in-production-please'). requireAuth (auth.js:22) calls verifyToken with the same secret and accepts it; requireAdmin then grants full /api/admin access — list/delete every owner, read every account's synced mirror (/api/admin/users/:id/data), reset plans, etc. They can equally forge an owner/accountant token for any account and pull that owner's entire business mirror via /api/sync/pull. This is a complete authentication bypass.
- **Root cause:** JWT_SECRET has an insecure default and there is no startup assertion that a real secret is configured when NODE_ENV==='production'. A misconfiguration (unset env var) degrades silently to a known-secret instead of refusing to boot.
- **Recommended fix:** In env.js (or server start()), abort boot when NODE_ENV==='production' and JWT_SECRET is unset or equals the placeholder: e.g. `if (env.NODE_ENV==='production' && (!process.env.JWT_SECRET || process.env.JWT_SECRET==='change-me-in-production-please')) throw new Error('JWT_SECRET must be set to a strong value in production');`. Remove the real-looking default from .env.example (use an empty/placeholder that fails the check) so a copied .env can't accidentally ship the known value.

#### Bootstrap admin seeded with weak default credentials (admin / admin123) with no production guard
- **Area:** `security` &nbsp;|&nbsp; **Severity:** CRITICAL
- **Location:** backend/src/bootstrap/seed.js:50-76 (ensureAdmin); defaults in backend/src/config/env.js:30-31
- **Scenario:** On first boot, runSeed()->ensureAdmin() creates an admin User from ADMIN_USERNAME/ADMIN_PASSWORD, which default to 'admin'/'admin123' (env.js:30-31, .env.example). If the operator deploys to production without overriding ADMIN_PASSWORD, the live admin panel at /admin is reachable with admin/admin123. Combined with no login rate limiting (see separate finding) and the open-by-default production URL, anyone can POST /api/auth/login {username:'admin',password:'admin123'} and obtain a real admin JWT (works even if JWT_SECRET *is* set), then read/delete every owner's mirror and manage all accounts and plans.
- **Root cause:** Insecure default credentials are applied unconditionally with no enforcement that they be changed in production, and ensureAdmin never rotates or warns about the default password.
- **Recommended fix:** Refuse to seed (or hard-fail boot) when NODE_ENV==='production' and ADMIN_PASSWORD is unset or equals 'admin123'. Require a strong ADMIN_PASSWORD via env in production and document it. Optionally log a loud warning in non-prod when the default is in use.

#### No rate limiting / lockout on POST /api/auth/login — online brute force of admin and owner passwords
- **Area:** `security` &nbsp;|&nbsp; **Severity:** HIGH
- **Location:** backend/src/routes/auth.js:24-33; backend/src/server.js:24-48 (no rate-limit or helmet middleware mounted)
- **Scenario:** server.js mounts only cors(), express.json, and morgan — there is no express-rate-limit, no account lockout, and bcrypt.compare runs on every attempt (authController.js:74). An attacker can issue unlimited /api/auth/login requests against the production server to brute-force any account, most damagingly the 'admin' account whose username is predictable and whose default password is weak (admin123). There is no throttle, captcha, IP ban, or failed-attempt counter anywhere in the request path.
- **Root cause:** Authentication endpoints are unthrottled and there is no lockout on repeated failures.
- **Recommended fix:** Add express-rate-limit (already a common dep) on /api/auth/login and /api/auth/register (e.g. 5-10 attempts/minute per IP) and consider a per-account failed-attempt backoff. Mounting helmet for security headers is also advisable.

#### BillingController price cache (currentPrices) not refreshed after a pull(), leaving Monthly Pricing screen and the collect-payment discount preview stale
- **Area:** `getx-ui-state` &nbsp;|&nbsp; **Severity:** MEDIUM
- **Location:** lib/controllers/sync_controller.dart:293-305; lib/controllers/billing_controller.dart:36-61; lib/views/widgets/collect_payment_dialog.dart:24
- **Scenario:** On device B, sign in / open the app on the same account; the Monthly Pricing tab is alive (IndexedStack keeps all 5 tabs mounted, main_screen.dart:29). On the dashboard tap 'update now' (SyncController.pull). pull() -> _reloadAppData() reloads ONLY DashboardController.loadStats and CoreController.loadBoards/loadSubscribers; it never calls BillingController.loadMonthPrice. So BillingController.currentPrice / currentPrices still hold the pre-pull values. The Monthly Pricing card and, more importantly, the collect-payment dialog's per-amp discount preview (collect_payment_dialog.dart:24 reads controller.currentPrices[category]) show stale prices until the user changes month/branch (the only triggers that reload billing).
- **Root cause:** _reloadAppData() does not include BillingController among the controllers it refreshes, and BillingController only reloads on its own ever(month)/ever(branch) hooks, neither of which fires on a pull. The dashboard money figures recompute from the DB directly, masking the stale billing cache, but billing-derived UI (pricing card + discount preview) desyncs from the just-pulled DB.

## Cross-cutting / architectural weaknesses

- **Conflict model.** The sync engine is delivery-correct (outbox + drain-after-success + pull outbox-suppression) but has **no causality/versioning**: apply is blind `ConflictAlgorithm.replace` (client) and blind upsert (server). Without a per-row `updatedAt`/version and a compare-on-apply, every offline-first multi-device scenario can lose updates or resurrect deletes. Several Critical/High findings are facets of this one gap.
- **Receipt numbering as an identity.** `receipt_no` is treated as a meaningful sequential key but is computed client-side and not unique across devices. Either make it **server-assigned** on push, or treat `uuid` as the only identity and `receipt_no` as a per-device display label that the server renumbers.
- **Trust boundary at `/api/sync/push`.** The mirror accepts whole rows verbatim with no validation. Combined with UI-only permission/branch enforcement, any authenticated accountant can write arbitrary entities/fields/branches into the owner's data. Add server-side schema/whitelist validation + branch/owner/permission scoping.
- **Financial-consistency invariant.** "Paid/Unpaid", "Collected", "Remaining", "Expected" are computed independently in **four** places (app Dashboard, app Reports, subscriber-detail card, backend dashboard) and the discount + per-branch-price handling already diverge. Define ONE canonical formula (coverage = paid + discount; remaining = expected − collected − discount) and make all four use it.
- **Pricing as live vs snapshot.** Due is computed from the *current* month price, so editing a month's price (or a subscriber's category) **retroactively rewrites history** for already-settled receipts. Use the receipt's `price_snapshot`/`category_snapshot` for historical due, and freeze settled months.
- **Scale ceiling.** Full-table loads (`limit:10000`), no pagination on Paid/Unpaid, whole-mirror pulls, and missing indexes mean refresh cost grows linearly with months of data — fine at demo size, painful after a year.

## Long-term architectural improvements
1. Add `updated_at` (and/or a monotonic version) to every synced row; server rejects/merges stale writes; client pull compares before overwrite. Consider soft-delete tombstones with timestamps instead of hard deletes.
2. Make `receipt_no` server-authoritative (assigned on first push) or drop its sequential meaning.
3. Server-side authorization + validation on `/api/sync/push` (entity whitelist, field whitelist, branch/owner/accountant scoping, type checks).
4. Centralize the billing math (one module/SQL view) consumed by app + backend; add an automated cross-consistency test.
5. Use `price_snapshot`/`category_snapshot` for all historical due; lock a month after first collection.
6. Honor the `since` watermark on pull; paginate all list/stat queries; add indexes on `receipts(branch_id, month, subscriber_id)`, `subscribers(branch_id, circuit_id, category)`, `monthly_prices(month, branch_id)`.
7. Harden logout-wipe: only wipe after a confirmed successful push AND only when re-pull is possible (online + sync-enabled); never on involuntary/offline logout.
8. Security baseline: mandatory `JWT_SECRET` (fail-fast if unset), no default admin in production, enforce `expiresAt`, enforce `maxDevices` server-side regardless of payload.
